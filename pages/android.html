Android Project

Notes API, 모임앱, 시놀 MBTI를 통해 Android UI 구현, API 연동, 상태 흐름 분리 경험을 쌓았다. 기능 나열보다 MVVM, 화면 책임 분리, 서버 연동 구조를 어떻게 다뤘는지에 초점을 맞췄다.

메모 앱, 모임앱, 인턴 프로젝트 화면을 묶어 Android UI 구성과 API 연동 경험을 한눈에 보여준다.

내 역할

화면 구현을 넘어 구조와 연동 흐름을 다뤘다

  • Android UI 구현
  • Flask/API 연동
  • 상태 흐름 관리
  • 기능별 화면 책임 분리

설계 포인트

기능보다 화면과 데이터 흐름 분리에 집중했다

  • MVVM 기반 화면 구조 정리
  • API 응답과 UI 상태 연결
  • 프로젝트별 기능 책임 분리
  • 모바일 사용 흐름 중심 UI 구성

프로젝트 개요

Android 페이지는 Notes API, 모임앱, 시놀 MBTI 프로젝트를 묶어 모바일 구현 경험을 정리한 페이지다. 메모 CRUD와 서버 연동, 커뮤니티형 화면 흐름, 인턴 프로젝트에서의 API 연동과 상태 관리까지 서로 다른 성격의 작업을 통해 화면 구조와 데이터 흐름을 다뤘다. 핵심은 기능을 많이 보여주는 것이 아니라, 모바일 UI와 API 연결을 어떤 구조로 정리했는지 드러내는 데 있다.

프로젝트 묶음과 핵심 기술

프로젝트 묶음

  • Notes API: Android 앱과 Flask 서버를 연결해 메모 CRUD와 텍스트 가공 흐름을 실험했다.
  • 모임앱: 모임 생성, 참여, 채팅 기능을 중심으로 커뮤니티 서비스 흐름을 구현했다.
  • 시놀 MBTI: 테스트 결과, 추천, 공유 흐름을 포함한 인턴 프로젝트에 참여했다.

핵심 기술

  • Kotlin 기반 Android UI 구현
  • MVVM과 상태 흐름 분리
  • Flask/API 연동과 데이터 처리 경험
설계와 구조

설계 관점

Android 프로젝트에서는 화면 구현보다 화면 상태와 데이터 요청이 서로 섞이지 않도록 나누는 데 신경 썼다. 기능마다 UI, 상태, API 연동 책임을 구분해 이후 수정과 확장이 가능하도록 구조를 정리했다.

구조 예시

app/
|-- data/
|-- domain/
|-- presentation/
|-- di/
`-- ui/

패키지 분리 기준을 먼저 세워 화면 코드와 데이터 처리 코드를 분리하는 방향으로 접근했다.

성과와 학습 포인트

성과

  • Android UI 구현과 서버 연동을 함께 다루는 경험을 쌓았다.
  • 프로젝트별로 다른 화면 흐름을 구조적으로 정리해보는 기준을 만들었다.
  • 모바일 앱에서 상태 관리와 API 연결을 분리해보는 실전 경험을 확보했다.

학습 포인트

  • MVVM 구조가 화면 수정과 기능 확장에 어떤 이점을 주는지 체감했다.
  • 서버 응답과 UI 상태를 연결하는 흐름이 모바일 경험에 큰 영향을 준다는 점을 배웠다.
  • 기능 구현보다 화면 책임을 어떻게 나누는지가 유지보수성에 더 중요하다는 기준을 얻었다.
