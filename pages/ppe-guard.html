PPE Guard

웹캠·CCTV 입력부터 PPE 탐지, OCR, 결과 저장·조회까지 한 흐름으로 연결한 온프레미스 안전 점검 시스템입니다.

입력, 탐지, OCR, 저장 흐름이 이어지는 실제 처리 화면입니다.

내 역할

  • 프로젝트 방향과 MVP 범위를 주도적으로 정리했습니다.
  • DB와 화면 흐름을 함께 설계해 입력부터 조회까지 구조를 묶었습니다.
  • YOLO·OCR 파이프라인과 판정 로직을 설계했습니다.
  • OCR 조건과 처리 단계를 분리해 병목을 줄였습니다.

기술 의사결정

  • 사내망 환경에 맞는 설치형 구조를 택했습니다.
  • 클라이언트 직결 대신 서버 중심 구조로 정리했습니다.
  • 영상 특성에 따라 track / predict 전략을 나눴습니다.
  • OCR은 필요한 구간에만 실행해 비용을 줄였습니다.

프로젝트 개요

현장 안전 점검은 탐지 기능보다 실제 운영 가능한 구조가 더 중요하다고 판단했습니다. 그래서 웹캠·CCTV 입력, 분석 처리, 결과 저장·조회가 한 흐름으로 이어지는 온프레미스 구조를 중심으로 기획했습니다. 그 결과 클라이언트, 서버, 저장소, AI 처리 단계를 분리한 PPE 점검 시스템을 구현했습니다.

핵심 기술과 아키텍처

핵심 기술

  • Hexagonal Architecture로 화면, 애플리케이션, 도메인, 인프라 책임을 분리했습니다.
  • Docker 기반 실행 구조로 설치와 배포 단계를 단순화했습니다.
  • YOLOv11과 EasyOCR을 단계별 파이프라인으로 나눠 운영 흐름에 맞췄습니다.

시스템 구조

PySide6 클라이언트가 입력과 조회를 담당하고, Flask 서버가 분석 요청과 결과 저장을 처리합니다. 분석 엔진은 YOLO 탐지와 OCR을 분리해 실행하며, 저장소는 SQLite 또는 MySQL로 구성했습니다.

project/

UI, 흐름 제어, 판정 규칙, 외부 연동 책임을 나눠 기능 확장이나 환경 변경 시 영향 범위를 줄였습니다.

트러블슈팅

  • 문제 YOLO와 OCR을 동시에 실행하면 CPU 사용량이 급격히 증가했습니다. 탐지 결과가 필요한 구간에만 OCR을 수행하도록 조정해 병목을 완화했습니다.
  • 문제 worker 수를 늘려도 처리량이 기대만큼 늘지 않았습니다. 병렬화보다 처리 조건과 단계 분리가 더 효과적임을 확인하고 흐름을 다시 설계했습니다.
성과와 검증 범위

성과

  • 입력부터 탐지, OCR, 저장, 조회까지 이어지는 통합 흐름을 완성했습니다.
  • 사내망 환경에서도 운영 가능한 설치형 구조를 구체화했습니다.
  • 클라이언트와 DB 직접 연결을 피하는 서버 중심 설계를 검증했습니다.

검증

  • 분석 요청과 결과 저장·조회가 실제 시나리오대로 연결되는지 확인했습니다.
  • 탐지와 OCR이 함께 동작하는 처리 흐름을 실제 영상 기준으로 테스트했습니다.
  • 예외 처리와 로그 흐름까지 점검해 운영 안정성을 확인했습니다.
