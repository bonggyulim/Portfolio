RareBridge

자연어 증상을 HPO 기준 데이터로 표준화해 희귀질환 후보를 추천하는 검색·탐색 서비스입니다.

증상 입력이 HPO 매핑과 질환 추천으로 이어지는 핵심 흐름입니다.

내 역할

  • 요구사항과 기능 명세를 정리하고 전체 방향을 설계했습니다.
  • HPO 기반 DB 구조와 데이터 적재 방식을 설계했습니다.
  • 자연어 입력부터 질환 조회·출력까지 핵심 흐름을 정의했습니다.
  • 기능별 구현 결과를 하나의 서비스 흐름으로 통합했습니다.

문제 해결 방식

  • 자유 입력을 HPO 기반 데이터로 표준화했습니다.
  • 입력 → 매핑 → 스코어링 → 추천 흐름을 설계했습니다.
  • Repository / Service Layer로 로직을 분리했습니다.
  • 정규화된 질환·HPO DB 기반으로 확장성을 확보했습니다.

프로젝트 개요

희귀질환은 증상이 겹쳐 초기 단계에서 필요한 정보를 빠르게 찾기 어렵다는 문제에서 출발했습니다. 그래서 증상을 그대로 검색하는 대신, 입력을 HPO로 표준화한 뒤 질환 후보를 추천하는 흐름으로 기획했습니다. 그 결과 입력, 표준화, 질환 탐색, 결과 제공이 하나의 데이터 흐름으로 이어지는 구조를 구현했습니다.

핵심 기술과 아키텍처

핵심 기술

  • FastAPI로 입력 처리, HPO 매핑, 질환 조회 흐름을 API 단위로 분리했습니다.
  • Supabase PostgreSQL에 질환·HPO 관계 데이터를 정규화해 적재했습니다.
  • 자연어 입력을 구조화하는 전처리 단계와 추천 로직을 분리해 유지보수를 쉽게 했습니다.

시스템 구조

프론트엔드는 입력과 결과 화면을 담당하고, 백엔드는 매핑과 스코어링, 추천을 처리합니다. 저장소는 질환·HPO 관계형 데이터를 기준으로 구성해 추천 로직과 데이터 적재를 분리했습니다.

구조 설계와 검증

구조 설계

project/
|-- frontend/
|-- backend/
|-- domain/
|-- dataset/
|-- docs/
`-- docker/

UI, 추천 로직, 데이터 적재를 분리해 기능 확장과 데이터 보강이 서비스 흐름을 깨지 않도록 구성했습니다.

검증

  • 정상 입력뿐 아니라 공백, 특수문자, 불완전 입력 케이스까지 테스트했습니다.
  • 단위 테스트와 통합 테스트를 통해 API 예외 처리와 추천 흐름을 확인했습니다.
  • 입력이 부족한 상황에서도 추천 흐름이 깨지지 않는지 중점적으로 검증했습니다.
트러블슈팅과 확장 방향

트러블슈팅

  • 문제 단위 테스트는 통과하지만 통합 흐름에서 예외 케이스가 500 오류로 이어졌습니다. 예외 타입과 응답 코드를 명확히 나눠 API 레이어에서 일관되게 처리했습니다.
  • 문제 자유 입력 기반 구조라 비정형 입력 안정성을 설명하기 어려웠습니다. 공백, 특수문자, 빈 입력 케이스를 포함한 검증 범위를 넓혀 신뢰성을 보강했습니다.

확장 방향

  • cursor 기반 페이지네이션 적용
  • 질환 스코어링 가중치 고도화
  • 결과 화면 정보 확장
